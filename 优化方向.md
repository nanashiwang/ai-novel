## 1. 现状摘要

经过 ~140 个提交（约 7 周）的推进，项目已经升级为：

**端到端可运行的 AI 小说生产平台**，含 6 层分层生成内核、动态画像（人物/世界/剧情线三链同构）、SSE 实时推送、全书 orchestrator、内容审查、监控告警、离线评测、多 agent 写作流水线。

所有 Sprint 1–14 的 P0/P1 闭环均已落地：

```
创建项目
  → 故事启动中心（创作偏好 / 方向预览 / preflight 检查）
  → StoryBible 生成（spec + characters[v0 motivation/arc/secret] + world_items + plot_threads）
  → 章节大纲生成（chapters；支持 batch + append + force_regenerate）
        └ Outline 后 fire-and-forget：refine motivation/arc/secret 到 pending
  → 场景拆分（scenes，按 chapter）
  → 场景写作（draft_versions，可选 single / multi-agent 流水线）
        └ 写完后 fire-and-forget：character/world/plot 三链推演 + moderation 审查 + ledger 校验
  → 审稿（continuity_issues） → 重写（基于 issue 修复）
  → 导出（Markdown / TXT，HTTP 流式下载，MinIO 抽象）

实时层：所有生成事件经 Redis pub/sub → /events SSE → 前端 useEventSource
评测层：python -m app.evals.cli run 跑固定数据集 + LLM judge 输出报告
```

整条链路：
- 每一步创建 `generation_jobs` 并预留 quota；失败自动 release + revert project status
- 每次模型调用写入 `model_calls`（含 prompt_key / version / metadata_json / cost）
- ContextBuilder v3 共 **13 段**（hard_constraints / task / characters / character_actions / style_samples / world_rules / world_actions / plot_threads / plot_actions / recent_scenes / arc_summaries / information_visibility / memory_recall），预算严格 sum=1.0
- ModelGateway 默认走真实 provider；fast 模式 stream=true 兼容中转网关
- Health/Ready/Startup 三探针；Prometheus + Grafana 告警栈；237 后端测试 + Playwright e2e 金路径

**C3 · 多 agent 写作（planner → drafter → stylist）**
- `app/services/writer/{planner,drafter,stylist}.py` 三子服务 + 三个 v1 prompt
- `BeatSheetContract` + `BeatItem`（每 beat 含 purpose/action/dialog_hint/reaction/target_words）

**C5 · 信息释放 ledger**

**C6 · Per-scene POV 锚定**

## 5. 关键技术决策回顾

| 决策点 | 选择 | 原因 |
|---|---|---|
| Job 状态机 | `created / quota_reserved / queued / running / succeeded / failed / cancelled` | 与 Temporal 内部状态分离，便于 UI 与权限校验 |
| Job 命名 | `generate_bible / generate_outline / generate_scene_plan / write_scene / audit_scene / rewrite_scene / full_novel / export_novel` | Sprint 1 冻结，含合同测试（test_contract_consistency） |
| ModelGateway 模式 | 默认走真实 provider；mock 已完全移除（f06324c） | 避免 mock/prod 行为分裂；通过 admin 配置切换 OpenAI/Anthropic + base_url |
| Quota 预留/结算 | reserve at job create → settle / release on job end → revert project status on failure | 24c42f1 + 5e922c9 + e79a0c3 |
| Workflow Retry | 显式 `RetryPolicy` per activity；业务异常标记 non-retryable（116ae63） | 区分系统错误（重试）vs 业务错误（直接失败） |
| Outline 章节上限 | 单次按 60 章分批；项目级上限 200 | 避免 LLM 输出爆掉；append 模式只生成缺失尾部 |
| 富文本格式 | content TEXT + content_format 列；选 markdown 而非 jsonb | 后端无需新解析依赖；导出 markdown 零转换；旧 'text' 数据零侵入 |
| 编辑器 markdown serializer | 自实现（基于 ProseMirror doc） | tiptap-markdown 第三方包年久失修；StarterKit 节点集固定，自写约 80 行可控 |
| AI 设定共创安全模型 | AI 仅出"修改提案"，用户应用才写库 | 避免对话直接覆盖生产数据；保留人工最终决策权 |
| 前端聚合文件 | 全部拆为独立 Page + 子组件 | SRP；改动冲突减少；tree-shaking 生效；类型从 lib/api 统一 |
| 动态人物画像 | 单表 `character_revisions` 统一三种来源（手动 / Copilot / AI 推演） | 简化数据模型；AI 推演产出 pending 待审核，不直接覆盖；ContextBuilder 仅读 applied，避免污染 |
| 人物字段分阶段生成 | Bible 阶段只给 motivation/arc/secret 的 v0 草稿；Outline 后基于三幕重写 | 早期缺材料强行生成秘密/弧光质量差；推迟到 outline 后能引用具体章节 evidence |
| SSE vs WebSocket | 单向事件流走 SSE（Redis pub/sub 后端） | 不需要双向通信；EventSource 浏览器内置；连接断开自动重连；同源 cookie 直接生效 |
| 全书生成 vs 单 scene | child workflow + BATCH_SIZE=3 + continue-as-new | 单 workflow 历史会爆掉（>200 事件就需 CAN）；按章独立 quota 结算便于失败精确重试 |
| world/plot revision 链同构 character | 单表 + 三种来源 + state machine | 用户已熟悉 character_revisions 的 UX；不重复造轮子；前端组件复用 |
| Moderation 不阻断 | fire-and-forget 走结构化日志，不写表 | 内容审查是观测项而非守门员；高风险样本由日志聚合（loki/elk）人工复核；YAGNI 不上 moderation_records 表 |
| Information ledger secret 永不入 prompt | 段只注入 partial/public；secret 留 LedgerService 守门 | 信息差是悬念的命脉；让 LLM 看到 secret 会被诱导提前揭示 |
| 多 agent 写作默认 single | 配置开关 `WRITER_PIPELINE_MODE` | multi 模式 3 倍 token 成本；正式默认前等 C1 评测体系给出胜出证据 |
| POV 隔离做软过滤 | ContextBuilder 在 prompt 拼装时过滤非 POV 角色的 secret/motivation/arc | 不动 ORM 数据；不引入 character.knowledge_state 复杂建模；过滤逻辑集中在一处便于调优 |
| 评测体系不暴露 HTTP | 内部 CLI 工具，输出 JSON 报告 | KISS：开发者面而非用户面；CI gate 直接消费报告；未来要 UI 再说 YAGNI |
